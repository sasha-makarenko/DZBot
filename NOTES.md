## Часть A. Решения (ADR)

### ADR-001 — Python 3.12, а не 3.14
- **Контекст.** В системе стоял Python 3.14.3 (по умолчанию). Нужно было выбрать версию под ML-стек.
- **Решение.** Поставить рядом Python 3.12 (через `winget install Python.Python.3.12`) и делать `.venv` проекта на нём. 3.14 не трогать.
- **Почему.** ML-библиотеки с нативным CUDA-кодом (torch, bitsandbytes, части accelerate) выходят под новый Python с задержкой в месяцы. На 3.14 уже были случаи: `pip install torch` ставился «успешно», но как CPU-only сборка → `cuda.is_available() == False`. Норма в ML: держать Python на 1–2 версии позади самой свежей. Стабильность > новизна.
- **Альтернативы.** 3.13 (тоже годится, в активной фазе); 3.14 (отклонено — риск отсутствия wheel'ов).
- **Следствия.** Все установки делаем в `.venv` на 3.12.

### ADR-002 — PyTorch сборка cu128 + проверка GPU реальной операцией
- **Контекст.** GPU RTX 5070 Ti — архитектура Blackwell, compute capability sm_120 (12, 0). Драйвер 577.00, CUDA 12.9 (на стороне драйвера). VRAM 15.9 ГБ.
- **Решение.** Ставить PyTorch с индекса cu128: `python -m pip install torch torchvision --index-url https://download.pytorch.org/whl/cu128`. Проверять GPU не только `is_available()`, но реальной операцией (умножение матриц на `cuda` + `synchronize()`).
- **Почему.** Обычный `pip install torch` на Windows тянет с PyPI CPU-версию (тихая ловушка). Индекс cu128 содержит ядра (kernels) под sm_120. Главное: `is_available() == True` **необходимо, но недостаточно** — при отсутствии ядер sm_120 проверка проходит, а реальная операция падает с `no kernel image is available for execution on the device`. Поэтому тест должен реально считать на GPU.
- **Результат проверки.** `2.11.0+cu128`, `CUDA available: True`, `RTX 5070 Ti`, capability `(12, 0)`, матрица посчиталась без ошибки.
- **Следствия.** Фундамент рабочий. Установки pip — всегда через `python -m pip` (см. Грабли G-002).

### ADR-003 — Старт с Qwen3-VL-4B-Instruct (bf16); 8B+4-bit отложить до Фазы 1
- **Контекст.** VRAM 16 ГБ. Линейка Qwen2.5-VL устарела, актуальна Qwen3-VL (размеры 4B/8B, варианты Instruct/Thinking), поддержана в свежем transformers.
- **Решение.** Стартовать с **Qwen3-VL-4B-Instruct в bf16, без квантизации**. Подъём до 8B+4-bit — в Фазе 1, и **по результатам eval**, а не заранее.
- **Почему (арифметика памяти).** Память ≈ параметры × байт_на_параметр. Точность: fp32=4 байта, fp16/bf16=2, 8-bit=1, 4-bit=0.5.
  - 8B × 2 = ~16 ГБ только веса → в 16 ГБ впритык, без запаса на вычисления/картинку/контекст (для 8B fp16 рекомендуют 24 ГБ+).
  - 4B × 2 = ~8 ГБ → влезает с большим запасом.
  - 8B × 0.5 (4-bit) = ~4 ГБ → влезает, но требует bitsandbytes (точка отказа, см. ADR-?/Грабли).
  Задача Фазы 0 — доказать, что конвейер живой, а не выжать максимум. Сначала самое маленькое, что надёжно заводится; качество измеряем, а не предполагаем.
- **Альтернативы.** Сразу 8B+4-bit (отклонено — риск утонуть в установке bitsandbytes на первом же шаге); вариант Thinking (отклонено — нужно прямое пояснение, не цепочка рассуждений).
- **Следствия.** bitsandbytes сейчас не ставим.

### ADR-004 — bitsandbytes: карта рисков и лестница отступления (на Фазу 1)
- **Контекст.** 4-bit квантизация для 8B пойдёт через bitsandbytes. Библиотека несёт свои CUDA-ядра (нужны под sm_120), исторически Linux-first, на Windows + новая CUDA капризна (были открытые issue по Blackwell/CUDA 12.9 на Windows).
- **Решение.** В Фазе 1 — сначала smoke-test (поставить свежий bitsandbytes, загрузить маленькую модель в 4-bit, один проход), и только потом 8B.
- **Типичные провалы.** (1) несовпадение версии CUDA → не находит ядра; (2) Windows не находит CUDA-DLL при импорте; (3) тихий CPU-fallback (работает, но дико тормозит).
- **Лестница отступления.** Свежий bitsandbytes → WSL2/Docker (под Linux связка предсказуемее) → другой формат квантизации (AWQ/GPTQ/GGUF) → остаться на 4B, если eval не покажет выигрыша 8B.
- **Принцип.** Решает измерение (eval), а не «как тяжело было поставить».

### ADR-005 — Docker: не сейчас, входит на этапе деплоя
- **Контекст.** Возник вопрос, нужен ли Docker.
- **Решение.** Сейчас — нет. Docker появляется на границе «инструмент для себя» → «инструмент, который запускается где-то ещё».
- **Почему.** Проект на одной машине, `.venv` изолирует зависимости, GPU работает нативно. Docker на Windows+GPU = доп. сложность (WSL2-бэкенд, NVIDIA Container Toolkit, проброс GPU) без текущей пользы.
- **Когда понадобится.** Фаза 2, если захочу шарить UI; и тем более облачный деплой (облака принимают приложение в виде контейнера). Также — как запасной путь для bitsandbytes (ADR-004).
- **Следствие.** Docker учим, когда проект упрётся в деплой, не раньше (иначе scope creep).

### ADR-006 — Голос: STT (вход) приоритетнее TTS (выход); порядок «ядро → голос»
- **Контекст.** Идея голосового взаимодействия (наблюдения: как люди общаются с LLM голосом; создатель OpenClaw наговаривает задачи).
- **Решение.** Голос — законная часть видения, с акцентом на **ввод (STT)**. Озвучка ответа (**TTS**) — опционально. Любой голос — строго после рабочего ядра.
- **Почему.** STT (говорю «поясни это» вместо печати) реально меняет UX. TTS (робот зачитывает конспект) слабее: длинный текст на слух тяжелее, формулу проще увидеть. Ловушка рассуждения: «эксперт делает X → X универсально» неверно — голос хорош для одних кусков (команды/намерения) и плох для других (чтение).
- **Следствие.** В roadmap: STT — Фаза 3 (надстройка над UI); TTS — опционально.

### ADR-007 — Двухканальный вывод: краткий голос + подробный конспект, одним промптом
- **Контекст.** Хочется, чтобы вслух было коротко (как репетитор), а в конспект писалось подробно.
- **Решение.** Один разбор → два представления: краткое устное пояснение + подробный `.md`-конспект. Реализовать **промптом** (модель возвращает два блока за один проход), код разводит их по каналам.
- **Почему.** Разные цели потребления → разный текст (понять сейчас vs хранить и перечитывать). Это «разделение по каналам вывода» — снова чистый промпт-инжиниринг, без второй модели.
- **Следствие.** Лежит в Фазе 3 (где появляется голос); пока строим только подробный канал (Фаза 0b).

### ADR-008 — Документация: blueprint (план) отдельно от NOTES (журнал)
- **Контекст.** Куда складывать знания, чтобы не потерялись с закрытием чата.
- **Решение.** Источник истины — файлы в репозитории. `blueprint.md` = план/текущее состояние (медленно меняется). `NOTES.md` = журнал решений и «почему» (растёт). Projects в Claude — удобная обёртка поверх этих файлов, но не источник истины.
- **Философия правок blueprint.** Разделы-описание актуализируем (будущее время → настоящее); разделы-указатели («следующий шаг») перезаписываем целиком; история и «почему» — в NOTES. Тест: «это правда прямо сейчас?».

## Часть B. Грабли и уроки (gotchas)

- **G-001 — Ловушка sm_120.** `is_available()` может вернуть True, а операция на GPU упасть с `no kernel image is available`. Урок: проверять GPU реальной операцией. (см. ADR-002)
- **G-002 — `pip.exe` блокирует Smart App Control** (Windows 11): ошибка `Application Control policy has blocked this file`. Решение: всегда `python -m pip install ...` (запуск pip как модуля через разрешённый python). Это и официально рекомендуемый способ.
- **G-003 — Активация venv в PowerShell** может падать с `running scripts is disabled`. Решение (один раз, только для пользователя): `Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned`.
- **G-004 — Python 3.12 в фазе security-only**: на странице релиза только source tarball, нет Windows-installer. Решение: `winget` (даёт сборку с установщиком) или взять 3.13 с python.org.
- **G-005 — `.gitignore` не убирает уже застейдженные файлы.** Если успел `git add` лишнее (например `.venv`) — очистить индекс: `git rm -r --cached .` (файлы на диске не трогает), затем `git add .` заново. Перед коммитом всегда `git status` — проверить, что `.venv/` и слайды не попали.
- **G-006 — git remote с плейсхолдером.** Однажды добавил remote с буквальным текстом `ТВОЙ-ЛОГИН/ИМЯ-РЕПО` → потом `add origin` ругался `already exists`. Решение: `git remote set-url origin <правильный-URL>`; проверять `git remote -v`.
- **G-007 — Предупреждение `LF will be replaced by CRLF`** — безобидно (стиль переносов строк Unix vs Windows), на код не влияет. Возникает из-за работы на Windows и Mac.

## Часть C. Ключевые концепции (что понял)

- **Тензор (tensor)** ≈ массив NumPy, но умеет жить и считаться на GPU (`device="cuda"`).
- **dtype / точность.** bf16 и fp16 — по 2 байта; bf16 (bfloat16) шире по диапазону, стабильнее для ML; Blackwell умеет нативно.
- **Формат `messages`** (`role` + `content` из типизированных блоков text/image) — **универсальный** способ обращения к LLM (OpenAI, Claude, Qwen). Мой текст = блок text, прикреплённое фото = блок image.
- **Конвейер инференса:** `processor` переводит текст+картинку в тензоры → `model.generate()` порождает токены → отрезаем токены запроса (`out[len(inp):]`) → `batch_decode` переводит токены обратно в текст.
- **`**inputs`** — распаковка словаря в именованные аргументы.
- **Метод / функция / атрибут:** точка `.` = «загляни внутрь»; скобки `()` = «вызови»; `y.shape` без скобок = атрибут (данные).
- **Git-цикл обновления:** `status → add → commit → push` (после первого `push -u` дальше просто `git push`); `add`/`commit` локальны, `push` — отправка на GitHub.
- ** `**import os` ** - предоставляет функции для взаимодействия с операционной системой. Он позволяет управлять файлами, папками и процессами, делая код кросс-платформенным и независимым от того, запущен он на Windows, macOS или Linux
