# AI-ассистент для разбора слайдов — полный Blueprint проекта

> Единый рабочий документ. Самодостаточен (заменяет предыдущий). Версия 2.
> Контекст: студент WU (Wirtschaftsinformatik, Data Science). Цель — AI engineering / AI consulting. Железо: ПК с GPU GeForce 5070 Ti (~16 ГБ VRAM), ОС Windows. Подход: локальная VLM через Hugging Face `transformers`, вывод в Markdown.

---

## 1. Идея в одну строку

Смотришь на слайд → VLM читает текст и формулы → объясняет концепцию простым языком → сохраняет структурированный конспект в Markdown.

## 2. Проблема и мотивация

- **Личная боль:** текущий разбор слайдов = скрин → отправить в LLM → прочитать пояснение → вручную перенести в Goodnotes. Качественно, но медленно и руками.
- **Вдохновение:** демо Bogdan Pirvu на ClawCon (агент через очки помогал с заданием).
- **Цель:** тот же качественный разбор, но автоматический и с готовым конспектом на выходе.
- **Рамка:** ценность — в автоматизации процесса, а не в очках. Очки/голос — поздние надстройки.

## 3. Что строим — и чего пока НЕ строим

**Строим (ядро):** изображение слайда → понимание через VLM → структурированный конспект в `.md`.

**Пока НЕ трогаем (сознательно, чтобы не распылять scope):**
- ❌ V-JEPA / видео-модели — не подходят для чтения текста. Это отдельный проект, если вообще.
- ❌ Очки Meta — поздний UX-слой (требуют native mobile + железо; публикация пока только для партнёров).
- ❌ Голос (TTS), живой захват с камеры — фазы 2–3, после рабочего ядра.

## 4. Архитектура (конвейер из 4 шагов)

1. **Захват (Capture).** Источник изображения слайда. Прогрессия: файл/скриншот (сейчас) → камера телефона (позже) → камера очков (намного позже).
2. **Понимание (Understanding).** VLM читает и интерпретирует слайд напрямую (текст, формулы, диаграммы).
3. **Пояснение (Explanation).** Модель генерирует (а) понятное объяснение и (б) структуру конспекта.
4. **Вывод (Output).** Сохранение конспекта в `.md` (позже — озвучка через TTS).

**Главная мысль:** основной «интеллект» проекта — не код, а **промпт**. Кода немного (загрузить модель, дать картинку, сохранить ответ). Качество разбора определяется тем, как сформулирован запрос. Промпт-инжиниринг + оценка качества — это ядро работы и самый ценный навык под цель.

## 5. MVP — Фаза 0 (первый и единственный текущий фокус)

Разбита на три маленьких шага, строим строго по очереди:

- **0a — VLM читает слайд.** Картинка `slide.png` → VLM → печатает пояснение простым языком. Цель: доказать, что ядро работает.
- **0b — структурируем вывод.** Тем же вызовом, но промпт заставляет модель вернуть структуру: *резюме → ключевые понятия → термины с определениями → пояснение*. Чистый промпт-инжиниринг.
- **0c — пишем в Markdown.** Структурированный ответ → аккуратный `.md`-файл (заголовок, секции). На выходе — готовый конспект.

**Финальный результат MVP:** запускаешь скрипт на изображении слайда — получаешь на диске `.md`-конспект с пояснением и структурой. Это уже «как у Bogdan, но в твоём контексте», минус железо.

## 7. Технический стек

- **Среда:** Windows + GPU (CUDA), Python, виртуальное окружение, VS Code.
- **Ядро:** `transformers` (Hugging Face), PyTorch с поддержкой CUDA, PIL/Pillow для изображений.
- **Модель (VLM):** класс 7–8B под ~16 ГБ VRAM. Кандидаты на момент написания: **Qwen3-VL (8B)** или **Qwen2.5-VL-7B-Instruct** — сильное чтение документов и формул. При нехватке памяти — квантизация в 4-bit (bitsandbytes) или модель поменьше (3–4B). Точный выбор подтверждается при первом запуске; VLM быстро обновляются — проверять актуальную версию на старте.
- **Вывод:** генерация Markdown средствами Python.
- **Фаза 2+:** TTS (системный или API), Gradio для UI.
- **Фаза 3–4:** камера телефона; затем Meta Wearables Device Access Toolkit (native mobile).

## 8. Окружение на Windows (что предстоит настроить)

- Python + виртуальное окружение (как на Mac, синтаксис активации чуть другой).
- **PyTorch со сборкой под CUDA** именно для свежей карты — критично. После установки проверять: `import torch; print(torch.cuda.is_available())` → должно быть `True`. Если `False` — модель тихо считается на CPU, чинить.
- Свежий драйвер NVIDIA.
- `transformers` и зависимости.
- Детальная пошаговая настройка — отдельным шагом при старте Фазы 0.

## 11. Риски и слабые стороны (честно)

- **Расползание scope** — главный убийца. Держать фазы, не прыгать в очки/реалтайм.
- **Технология под резюме** (соблазн вставить JEPA) — убрано сознательно.
- **Точность VLM** на плотных слайдах с формулами — измерять (eval), не предполагать.
- **Память GPU:** 16 ГБ ограничивает размер модели; решается квантизацией / контролем разрешения.
- **Приватность/запись на лекциях** — учитывать на поздних фазах (политики универа, люди в кадре).
